# TFT-анализ спроса сети АЗС

> Авторская проектная работа. Автор: **ВПИШИ_СВОЕ_ФИО_ЗДЕСЬ**.
> Использование, демонстрация или сдача без указания автора запрещены.

Проект решает задачу анализа и прогнозирования спроса сети АЗС с помощью Temporal Fusion Transformer. Основной акцент в dashboard сделан на прогноз, сравнительный анализ топлива/магазина, переобучение модели и рекомендации.

## Структура проекта

```text
tft_predict/
  dashboard.py              Streamlit dashboard
  train.py                  обучение TFTModel
  predict.py                инференс сохраненной модели
  preflight.py              быстрая проверка данных
  data_preparation.py       подготовка признаков и Darts TimeSeries
  recommendations.py        правила рекомендаций
  config.py                 пути, цели, ковариаты, статические признаки
  README.md                 единая инструкция
  AUTHOR.md                 авторская маркировка
  requirements.txt          зависимости
  data/                     исходные CSV
  docs/                     задание и описание данных
  artifacts/                сохраненная модель, препроцессоры, manifest
  scripts/legacy/           старый вспомогательный пример
```

## Запуск

Рекомендуемое окружение:

```powershell
conda activate dl
```

Проверка данных:

```powershell
python preflight.py
```

Запуск dashboard:

```powershell
streamlit run dashboard.py
```

Обучение из терминала:

```powershell
python train.py --data data/detailed_data.csv --metadata data/stations_metadata.csv --output artifacts
```

Проверка пайплайна без обучения:

```powershell
python train.py --dry-run --station-limit 2
```

Отдельный прогноз из терминала:

```powershell
python predict.py --station-id 0 --horizon 24 --output forecast.csv
```

## Левая панель

`АЗС` выбирает всю сеть или одну станцию. В режиме всей сети прогноз и графики агрегируются по всем АЗС.

`Целевая метрика` задает основной ряд для вкладки прогноза и части сравнительных графиков. Доступны суммарные продажи топлива, продажи отдельных видов топлива, общая выручка магазина и категории магазина.

`Агрегация` управляет историческими графиками: час, день или неделя.

`Горизонт прогноза` выбирает длину будущего прогноза: 24, 72 или 168 часов.

`Период` ограничивает исторический срез данных. Все сравнительные вкладки, KPI и рекомендации считаются по этому выбранному периоду.

## Верхние карточки

Карточки сверху не являются выводом нейросети. Это быстрые фактические суммы за последние 14 дней выбранного среза:

- топливо;
- магазин;
- трафик;
- число АЗС;
- доля часов с промо или рекламой.

Они нужны как краткий контекст перед прогнозом и рекомендациями.

## Вкладка `Прогноз`

Главная вкладка проекта. Она показывает, что делает сохраненная TFT-модель или baseline, если модель недоступна.

### Основной график прогноза

На графике есть:

- фактический ряд за последние 21 день;
- прогноз на выбранный горизонт;
- подпись области прогноза.

Если доступны `artifacts/tft_station_model.pkl`, `artifacts/preprocessors.pkl` и `artifacts/manifest.json`, используется сохраненная TFT. Если инференс не сработал, dashboard явно показывает baseline.

### Статус модели

Под графиком выводится карточка:

- `TFT-прогноз построен`, если используется сохраненная модель;
- предупреждение об ошибке inference, если артефакты есть, но запуск не завершился;
- baseline-сообщение, если модель не обучена.

### Блок `Контекст спроса и конкуренты`

Этот блок нужен для объяснения факторов, которые связаны со спросом.

Режим `Контекст спроса` позволяет выбрать, что сравнить:

- праздники и обычные дни;
- погодные условия;
- промо и рекламу;
- тип дороги АЗС.

Для выбранного среза показываются:

- столбчатый график среднего значения выбранной метрики за час;
- таблица со средним и числом наблюдений.

Режим `Конкуренты АЗС` сравнивает собственные цены АЗС с конкурентными ценами:

- выбор топлива;
- график разницы `своя цена - цена конкурентов`;
- таблица по каждой АЗС;
- оценка: дешевле, на уровне или дороже конкурентов;
- детализация динамики цен для выбранной АЗС.

Важно: этот блок показывает исторические сравнения. Это не причинный uplift и не объяснение весов TFT.

## Вкладка `Сравнение: топливо`

Вкладка нужна для анализа продаж топлива в целом и по видам.

### График `Продажи по видам топлива`

Показывает исторические продажи:

- `sales_AI92`;
- `sales_AI95`;
- `sales_AI98`;
- `sales_DT_EURO`;
- `sales_DT_TANEKO`;
- `sales_DT_SUMMER`;
- `sales_DT_WINTER`.

Если прогноз построен, он продолжается пунктиром на том же графике. Это позволяет визуально сравнить факт и прогноз.

### Таблица итоговых литров

Показывает сумму продаж каждого вида топлива за выбранный период. По ней можно быстро объяснить, какой вид топлива дал основной исторический объем.

### Таблица прогнозного состава

Показывает прогнозный объем каждого вида топлива на выбранном горизонте и его долю. Эта таблица используется для рекомендаций по запасам.

### `Сценарный анализ топлива`

Это интерактивный конструктор исторического сценария. Он отбирает реальные часы из данных и показывает, как в таких условиях отличались продажи.

Можно фильтровать:

- погоду;
- температуру;
- осадки;
- видимость;
- ветер;
- снег, дождь, туман;
- сезон;
- час суток;
- выходной;
- праздник;
- час пик;
- ночь;
- рекламу;
- канал рекламы;
- промо топлива;
- промо магазина;
- промо кафе;
- доминирующий тип транспорта;
- общий трафик или отдельный тип транспорта;
- тип дороги;
- направление;
- размер населенного пункта;
- удаленность от города;
- число конкурентов в 5 км;
- число колонок;
- площадь магазина;
- наличие мойки, кафе, магазина.

После выбора сценария показываются:

- сколько часов попало в сценарий;
- сколько АЗС осталось;
- среднее значение за час;
- отклонение от всего выбранного среза;
- сумма продаж в сценарии;
- динамика сценария против всего среза;
- средний состав продаж топлива;
- выбранный разрез сравнения;
- таблица средних по группам;
- scatter-график по выбранному числовому признаку;
- состав трафика;
- рейтинг АЗС по среднему спросу в сценарии.

Пример защиты: можно выбрать дождь, рекламу, федеральную трассу, высокий трафик легковых и показать, как в таком историческом сценарии менялись продажи топлива.

## Вкладка `Сравнение: магазин`

Вкладка анализирует выручку магазина и категории сопутствующих товаров.

### График выручки магазина

Показывает фактическую выручку `shop_total_revenue`. Если прогноз доступен, продолжает график пунктиром на горизонт прогноза.

### Диаграмма категорий

Показывает доли категорий магазина:

- напитки;
- закуски;
- автотовары;
- кофе;
- табак.

### Таблица прогнозного состава магазина

Показывает прогноз по категориям магазина и долю каждой категории на выбранном горизонте.

### `Сценарный анализ магазина`

Работает так же, как сценарный анализ топлива, но целевые метрики относятся к магазину:

- `shop_total_revenue`;
- категории магазина.

Можно ответить на вопросы:

- как менялась выручка магазина в праздники;
- как отличаются продажи кофе при утренних часах;
- что происходит с категориями при рекламе;
- как влияет наличие кафе;
- как связаны трафик и магазинная выручка;
- какие АЗС дают максимальную среднюю выручку в выбранном сценарии.

## Вкладка `Обучение`

Вкладка оставлена в основном интерфейсе, потому что она показывает прикладной пайплайн переобучения.

### Загрузка данных

Можно загрузить:

- новые детальные данные CSV;
- новые metadata CSV.

Если файлы не загружены, используются стандартные:

- `data/detailed_data.csv`;
- `data/stations_metadata.csv`.

Dashboard показывает предпросмотр CSV и проверяет наличие обязательных колонок:

- `timestamp`;
- `station_id`;
- `station_name`;
- хотя бы одна целевая колонка из `config.py`.

### Гиперпараметры

Можно настроить:

- `epochs`;
- лимит АЗС для быстрого smoke-теста;
- `batch_size`;
- `input_chunk`;
- `output_chunk`;
- `hidden_size`;
- `lstm_layers`;
- `attention_heads`;
- `dropout`;
- `learning_rate`;
- `weight_decay`;
- `seed`;
- `precision`.

### Сохранение результата

Если включить `Сохранять результат в рабочую папку artifacts`, успешное обучение заменит рабочую модель, которую использует вкладка `Прогноз`.

Если галочка выключена, обучение сохраняется в отдельный run внутри:

```text
artifacts/training_runs/
```

### Кнопки

`Проверить подготовку данных` запускает dry-run: данные читаются, признаки готовятся, manifest сохраняется, но TFT не обучается.

`Запустить обучение` запускает реальное обучение в отдельном процессе.

### Мониторинг

Вкладка показывает:

- список последних запусков;
- статус;
- автообновляемый прогресс выполнения;
- текущую эпоху;
- текущий batch;
- последний `train_loss`;
- `val_loss`, если он уже появился в TensorBoard;
- график динамики `train_loss` / `val_loss`;
- параметры запуска;
- сырой лог в свернутом блоке.

Мониторинг обновляется автоматически примерно раз в 5 секунд. Поэтому после нажатия `Проверить подготовку данных` или `Запустить обучение` не нужно вручную обновлять страницу: карточки и график подтягивают свежие значения из `train.log` и TensorBoard events внутри папки выбранного run-а.

## Вкладка `Рекомендации`

Рекомендации формируются правилами по факту и прогнозу. Это не отдельный текстовый вывод TFT.

Используются такие проверки:

- сравнение последних 14 дней продаж топлива с предыдущими 14 днями;
- ситуация, когда трафик растет быстрее продаж;
- прогнозный состав спроса по видам топлива;
- ведущая категория магазина по фактической выручке.

Прогнозная рекомендация по запасу топлива зависит от TFT-прогноза или baseline. Остальные рекомендации основаны на исторических правилах.

## Скрытые вспомогательные вкладки

Чтобы dashboard выглядел аккуратно на защите, вспомогательные EDA-вкладки скрыты. Они не удалены из кода и включаются флагами в начале `dashboard.py`:

```python
SHOW_OVERVIEW_TAB = False
SHOW_FACTOR_TABS = False
SHOW_TRAINING_TAB = True
SHOW_PREFLIGHT_SUMMARY = False
```

`SHOW_OVERVIEW_TAB = True` вернет общий обзор данных.

`SHOW_FACTOR_TABS = True` вернет отдельные вкладки акций, трафика и цен.

`SHOW_PREFLIGHT_SUMMARY = True` вернет технический JSON-summary в рекомендациях.

## Ограничения

Dashboard показывает исторические сравнения факторов, но не доказывает причинный эффект.

Вкладки сценарного анализа работают по фактическим историческим часам. Они не генерируют новый TFT-прогноз при вручную заданной погоде или рекламе.

Если добавить новый признак в модель, ее нужно переобучить, потому что изменится набор входных ковариат.
