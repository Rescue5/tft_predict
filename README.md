# TFT-анализ спроса сети АЗС

Этот проект закрывает задание по Temporal Fusion Transformer для сети АЗС на данных `detailed_data.csv` и `stations_metadata.csv`.

## Что требовало задание

В файле `Задание.docx` сформулированы основные требования:

1. Провести TFT-анализ сети АЗС по выданным данным и построить dashboard.
2. Обучить TFT, сохранить модель и дать возможность использовать сохраненную модель повторно.
3. Анализировать продажи топлива в целом и по видам топлива с учетом рекламы, трафика по типам транспорта, акций, цен конкурентов и других факторов из датасета.
4. Строить прогнозы продаж топлива и сопутствующих товаров.
5. Выдавать рекомендации и объединить анализ, прогнозы и рекомендации в dashboard.

`описание данных.docx` задает структуру входных данных: 25 АЗС, почасовой ряд за 2023 год, продажи 7 видов топлива, выручку 5 категорий магазина, трафик по типам транспорта, акции, рекламу, погоду, цены, цены конкурентов, временные и статические признаки АЗС.

## Что реализовано

- `data_preparation.py` загружает данные, проверяет целевые колонки, добавляет временные признаки, кодирует категории и собирает Darts `TimeSeries`.
- `train.py` обучает Darts `TFTModel`, сохраняет модель, препроцессоры, manifest и чекпоинты.
- `predict.py` загружает сохраненные TFT-артефакты и строит прогноз до 168 часов. Для режима без модели есть baseline по часовому профилю последних данных.
- `recommendations.py` формирует рекомендации правилами по факту и прогнозу.
- `dashboard.py` показывает анализ факторов спроса, прогнозы и рекомендации в Streamlit.
- `artifacts/` уже содержит сохраненную TFT-модель, препроцессоры, manifest и чекпоинты.

Проверка данных в текущем проекте проходит: `219000` строк, `25` АЗС, период `2023-01-01 00:00:00` - `2023-12-31 23:00:00`, по `8760` строк на станцию, без дублей `station_id + timestamp` и без пропусков в числовых полях после подготовки.

## Запуск

Для этого проекта удобно использовать уже настроенное окружение `conda dl`:

```powershell
conda activate dl
python preflight.py
streamlit run dashboard.py
```

После запуска Streamlit откройте адрес, который он напечатает в терминале. Dashboard не обучает модель при открытии.

Для обучения или переобучения:

```powershell
conda activate dl
python train.py --data detailed_data.csv --metadata stations_metadata.csv --output artifacts
```

Для проверки сборки рядов без обучения:

```powershell
python train.py --dry-run --station-limit 2
```

Для отдельного прогноза сохраненной TFT-моделью:

```powershell
python predict.py --station-id 0 --horizon 24 --output forecast.csv
```

## Как пользоваться dashboard

Слева задаются фильтры:

- `АЗС`: анализ всей сети или одной станции.
- `Целевая метрика`: суммарные продажи топлива, продажи отдельного топлива, общая выручка магазина или отдельная категория товаров.
- `Агрегация`: час, день или неделя для исторических графиков.
- `Горизонт прогноза`: 24, 72 или 168 часов.
- `Период`: двусторонний слайдер исторического интервала.

Статус в верхней части показывает, используется ли сохраненная TFT. Если артефакты доступны, вкладка `Прогноз` строит настоящий TFT-прогноз. Если артефактов нет или inference упал, dashboard явно переключается на baseline.

Карточки сверху - это не вывод нейросети. Это быстрые суммы по фактическим данным за последние 14 дней текущей выборки: продажи топлива, выручка магазина, трафик, число АЗС в срезе и доля часов с промо или рекламой.

## Вкладки dashboard

| Вкладка | Что показывает | Практический смысл |
| --- | --- | --- |
| `Обзор данных` | Историческую динамику выбранной метрики, тепловую карту по часу и дню недели, рейтинг АЗС, структуру топлива | Быстрый EDA по факту. Это бывший "Ситуационный обзор сети", а не отдельный алгоритм управления сетью |
| `Прогноз` | Факт за последние 21 день и будущий ряд по выбранной целевой метрике | Главная часть задания про прогноз. В режиме `Вся сеть` прогноз агрегируется по всем АЗС |
| `Топливо` | Продажи по видам топлива: факт сплошными линиями, прогноз пунктиром, итоговую таблицу литров и прогнозный состав спроса | Прямая проверка анализа топлива в целом и по типам. Forecast-часть делает рекомендацию по запасу проверяемой |
| `Магазин` | Выручку магазина с продолжением прогнозом, доли категорий и прогноз выручки категорий | Анализ и прогноз товаров без смешения с факторами спроса |
| `Акции и реклама` | Активность промо/рекламы и средние показатели при `ad_active` | Проверка факторов из задания. Это описательное сравнение, не доказательство причинного эффекта |
| `Трафик` | Трафик по типам транспорта и связь трафика с продажами топлива | Проверка фактора трафика из задания |
| `Цены` | Собственные цены, цены конкурентов и корреляции факторов спроса | Проверка ценовой и конкурентной части задания |
| `Рекомендации` | Карточки решений и preflight summary | Рабочая часть поддержки решений, но логика рекомендаций правиловая |

Вкладки `Акции и реклама`, `Трафик` и `Цены` нужны для показа факторов, прямо перечисленных в задании. Поэтому они не удалены. Их можно скрыть для более короткой демонстрации, выставив `SHOW_FACTOR_TABS = False` в начале `dashboard.py`.

Пустых декоративных вкладок сейчас нет. Разница в роли вкладок такая: `Прогноз` реально использует TFT при наличии артефактов, `Рекомендации` используют факт плюс прогноз, остальные вкладки в основном объясняют данные и факторы, на которых строится задача.

## Что именно делают рекомендации

Рекомендации не являются отдельным текстовым выводом TFT. Код `recommendations.py` проверяет правила:

- сравнивает последние 14 дней продаж топлива с предыдущими 14 днями;
- проверяет ситуацию, когда трафик растет быстрее продаж;
- по прогнозу суммирует будущий спрос по видам топлива, выбирает вид с наибольшим прогнозным объемом и показывает его долю среди прогнозируемых видов топлива;
- по фактической выручке последних 14 дней выделяет ведущую категорию магазина.

Значит, ответ на вопрос "рекомендации основаны на ответах сети?" такой: частично. Прогнозная рекомендация по запасу топлива зависит от прогноза TFT или baseline, а остальные рекомендации основаны на правилах по историческим данным. Эта доля не является причинным эффектом вида топлива: это доля его прогнозного объема в сумме прогнозов `sales_*` на выбранном горизонте.

## Ограничения

- Dashboard не показывает важности признаков TFT и не делает причинный анализ рекламных кампаний.
- Вкладка `Акции и реклама` сравнивает наблюдения в датасете, но не оценивает uplift акции.
- Прогноз TFT строится по целевым метрикам из `config.py`: топливо, виды топлива, общая выручка магазина и категории магазина.
